# CAPMAAS end-to-end tests

These tests follow the pattern used by the upstream Cluster API providers (e.g.
[cluster-api-provider-vsphere](https://github.com/kubernetes-sigs/cluster-api-provider-vsphere/tree/main/test/e2e)):
a single flat suite that

1. boots a local [kind](https://kind.sigs.k8s.io/) management cluster,
2. builds a local `clusterctl` repository from the providers declared in
   [`config/maas.yaml`](./config/maas.yaml) (CAPI core + kubeadm bootstrap/control-plane from
   upstream releases, and the **MAAS infrastructure provider built from this working tree**),
3. installs the providers into the management cluster, and
4. drives the workload-cluster lifecycle by reusing the shared spec library in
   `sigs.k8s.io/cluster-api/test/e2e`.

The whole package is guarded by the `e2e` build tag, so it is excluded from `make test` and the
default build.

## Specs

| Spec | Source | What it does |
|------|--------|--------------|
| `QuickStartSpec` | `quick_start_test.go` | Provisions a MAAS workload cluster from the default flavor and waits for it to become healthy (control plane up, CNI applied, nodes Ready). |
| `MachineDeploymentScaleSpec` | `md_scale_test.go` | Scales the worker `MachineDeployment` out and back in. |

Both are thin wrappers over the shared `capi_e2e` specs.

## CNI

The shared specs do **not** install a CNI. As with the other CAPI providers, Calico is delivered
by a `ClusterResourceSet` baked into the cluster-template flavor:

- `data/cni/calico/calico.yaml` — the Calico manifest, inlined into a `ConfigMap` via the
  `${CNI_RESOURCES}` placeholder (the framework reads the path in the `CNI` config variable).
- `data/infrastructure-maas/kustomize/main/cluster-resource-set.yaml` — the `ConfigMap` +
  `ClusterResourceSet`.
- `data/infrastructure-maas/kustomize/main/cni-label-patch.yaml` — adds the `cni` label to the
  `Cluster` so the CRS selects it.

This requires the `EXP_CLUSTER_RESOURCE_SET` feature gate, which is enabled in `config/maas.yaml`.

## Layout

```
test/e2e/
├── config/maas.yaml                         # E2EConfig: providers, variables, intervals
├── data/
│   ├── cni/calico/calico.yaml               # Calico manifest (applied via ClusterResourceSet)
│   ├── shared/main/metadata.yaml            # CAPI core/bootstrap/control-plane metadata
│   └── infrastructure-maas/
│       ├── kustomize/main/                  # kustomize SOURCES for the default flavor
│       │   ├── kustomization.yaml
│       │   ├── cluster-template.yaml        # base resources (${...} clusterctl vars)
│       │   ├── cluster-resource-set.yaml    # CNI CRS + ConfigMap
│       │   └── cni-label-patch.yaml         # cni label on the Cluster
│       └── main/
│           ├── cluster-template.yaml        # GENERATED by `make generate-e2e-templates`
│           └── metadata.yaml                # MAAS provider metadata
├── e2e_suite_test.go                        # suite harness (kind bootstrap, clusterctl init)
├── quick_start_test.go
└── md_scale_test.go
```

To regenerate the flavor after editing the kustomize sources:

```sh
make generate-e2e-templates
```

## Prerequisites

- Docker and `kind` for the management cluster.
- A reachable **MAAS deployment** with bare-metal or LXD hosts available for allocation, plus a
  deployable OS image that includes the target Kubernetes version.

## Required configuration

The MAAS-specific values in `config/maas.yaml` are placeholders. clusterctl also reads variables
from the OS environment, so export the values matching your MAAS deployment before running:

```sh
export MAAS_DNS_DOMAIN=maas
export CONTROL_PLANE_MACHINE_IMAGE=u-2204-0-k-1320-0
export WORKER_MACHINE_IMAGE=u-2204-0-k-1320-0
export CONTROL_PLANE_MACHINE_RESOURCEPOOL=default
export WORKER_MACHINE_RESOURCEPOOL=default
export KUBERNETES_VERSION=v1.32.0
# ... plus any MAAS endpoint / API-key env consumed by the controller's credentials
```

## Running

```sh
# Build the controller image, generate templates, and run the full suite.
make test-e2e

# Run only a subset by focus string or label.
make test-e2e GINKGO_FOCUS="QuickStart"
make test-e2e GINKGO_LABEL_FILTER="PR-Blocking"

# Reuse an existing management cluster / keep resources for debugging.
make test-e2e USE_EXISTING_CLUSTER=true SKIP_RESOURCE_CLEANUP=true
```

Artifacts (kind logs, controller logs, resource dumps) are written to `_artifacts/`.

## Validating without infrastructure

The spec tree can be validated (compiles, specs register, config/templates parse) without Docker
or MAAS using a Ginkgo dry-run:

```sh
go run github.com/onsi/ginkgo/v2/ginkgo -tags=e2e --dry-run -v ./test/e2e/... -- \
  -e2e.config="$(pwd)/test/e2e/config/maas.yaml" \
  -e2e.artifacts-folder="$(pwd)/_artifacts"
```
